# 历史记录洪水问题修复自测文档

## 1. 问题背景

### 1.1 Bug 现象

缩放（useResize）和多选拖拽（DndContext 中批量拖拽）操作在每次 mousemove 事件中都会调用 pushHistory 记录中间状态，导致撤销缓冲区（MAX_HISTORY_LENGTH = 50）在几秒钟内被填满。

用户体验问题：
- 拖拽或缩放结束后，按一次 Ctrl+Z 只能回退一个像素的变化
- 需要按 50 次 Ctrl+Z 才能回到操作前的状态
- 撤销/重做功能几乎不可用

### 1.2 问题根因

**问题触发路径**：

```
缩放/拖拽操作开始
     ↓
mousemove 事件触发（每帧 60+ 次）
     ↓
updateComponent 被调用
     ↓
pushHistory 记录历史
     ↓
history 数组快速增长
     ↓
MAX_HISTORY_LENGTH = 50 被填满
     ↓
历史记录被挤出
     ↓
用户无法撤销到操作前状态
```

**两个层面的问题**：

1. **Store 层**：`pushHistory` 每次调用都会记录一条历史记录
   - 没有批量操作的概念
   - 每次 `updateComponent` 都会触发历史记录

2. **组件层**：缩放和拖拽操作没有使用批量模式
   - `useResize.ts`：每次 mousemove 都调用 `updateComponent`
   - `DndContext.tsx`：拖拽过程中多次调用 `updateSelectedComponents`

## 2. 修复内容

### 2.1 Store 层新增批量历史记录支持 (`src/store/useBuilderStore.ts`)

**新增状态**：

```typescript
isInHistoryBatch: boolean;           // 是否处于批量历史记录模式
batchStartComponents: ComponentSchema[] | null;  // 批量操作开始时的组件状态
```

**新增方法**：

| 方法名 | 功能描述 |
|-------|---------|
| `beginHistoryBatch()` | 开始批量模式，保存当前组件状态快照 |
| `endHistoryBatch()` | 结束批量模式，比较开始和结束状态，如果有变化则记录 **1 条** 历史 |
| `cancelHistoryBatch()` | 取消批量模式，不记录任何历史 |

**修改 `pushHistory`**：

```typescript
// 修复前：无条件记录历史
pushHistory: (previousComponents, nextComponents) => {
  const { history, currentIndex, selectedComponentId, selectedComponentIds } = get();
  const newHistory = history.slice(0, currentIndex + 1);
  // ... 记录历史
};

// 修复后：批量模式下不记录历史
pushHistory: (previousComponents, nextComponents) => {
  const { history, currentIndex, selectedComponentId, selectedComponentIds, isInHistoryBatch } = get();

  if (isInHistoryBatch) {
    return;  // ✅ 批量模式下直接返回，不记录历史
  }

  const newHistory = history.slice(0, currentIndex + 1);
  // ... 记录历史
};
```

**`endHistoryBatch` 核心逻辑**：

```typescript
endHistoryBatch: () => {
  const { isInHistoryBatch, batchStartComponents, components, pushHistory } = get();

  if (!isInHistoryBatch || !batchStartComponents) {
    set({ isInHistoryBatch: false, batchStartComponents: null }, false, 'endHistoryBatch_no_op');
    return;
  }

  // 比较开始状态和结束状态
  const shouldSave =
    JSON.stringify(batchStartComponents) !== JSON.stringify(components);

  // 只有状态变化时才记录历史
  if (shouldSave) {
    pushHistory(batchStartComponents, components);  // ✅ 只记录 1 条历史
  }

  set({ isInHistoryBatch: false, batchStartComponents: null }, false, 'endHistoryBatch');
};
```

### 2.2 缩放操作修复 (`src/hooks/useResize.ts`)

**修改前**：

```typescript
const handleResizeStart = useCallback((e: React.MouseEvent, handle: ResizeHandle) => {
  // 设置缩放状态...
  // 没有批量历史记录保护
}, []);

const handleResizeEnd = useCallback(() => {
  // 重置缩放状态...
  // 没有结束批量历史记录
}, []);
```

**修改后**：

```typescript
const { 
  updateComponent, 
  beginHistoryBatch,    // ✅ 新增
  endHistoryBatch,      // ✅ 新增
  cancelHistoryBatch    // ✅ 新增
} = useBuilderStore();

const handleResizeStart = useCallback((e: React.MouseEvent, handle: ResizeHandle) => {
  // 设置缩放状态...
  beginHistoryBatch();  // ✅ 开始批量模式
}, [beginHistoryBatch]);

const handleResizeEnd = useCallback(() => {
  // 重置缩放状态...
  endHistoryBatch();    // ✅ 结束批量模式，只记录 1 条历史
}, [endHistoryBatch]);

const handleResizeCancel = useCallback(() => {
  // 重置缩放状态...
  cancelHistoryBatch();  // ✅ 按 Escape 取消，不记录历史
}, [cancelHistoryBatch]);

// Escape 键取消使用 handleResizeCancel
const handleKeyDown = (e: KeyboardEvent) => {
  if (e.key === 'Escape') {
    handleResizeCancel();  // 不记录历史
  }
};
```

  if (isCanvasItem(activeIdStr) || isSortableItem(activeIdStr)) {

## 3. 测试场景

### 场景 1：缩放操作后单次撤销验证

#### 测试步骤

1. **准备测试环境**
   - 启动开发服务器 `npm run dev`
   - 访问编辑器页面 `/builder`
   - 确保画布上至少有一个组件（如果没有，添加一个文本组件）

2. **记录初始状态**
   - 选中组件
   - 记录组件的初始位置（x, y）和尺寸（width, height）
   - 注意：此时「撤销」按钮可能是禁用状态

3. **执行缩放操作**
   - 将鼠标移到组件右下角的缩放手柄上
   - 按住鼠标左键并拖动
   - 拖动至少 100 像素（确保有明显的尺寸变化）
   - 松开鼠标左键结束缩放

4. **验证历史记录**
   - 观察「撤销」按钮是否变为启用状态
   - 只应该点击 **1 次** 就能撤销整个缩放操作

5. **执行撤销**
   - 点击「撤销」按钮，或按 `Ctrl+Z`
   - 观察组件是否回到缩放前的尺寸

6. **验证撤销效果**
   - 组件尺寸应与缩放前完全一致
   - 组件位置可能根据缩放手柄不同而变化，但尺寸应该恢复

7. **执行重做（可选）**
   - 点击「重做」按钮，或按 `Ctrl+Y`
   - 观察组件是否回到缩放后的尺寸

#### 预期结果

| 检查项 | 预期状态 |
|-------|---------|
| 缩放后 `canUndo` | true（启用） |
| `history` 长度增长 | +1（不是 +N） |
| 按 **1 次** `Ctrl+Z` | 组件回到缩放前尺寸 |
| 按 **1 次** `Ctrl+Y` | 组件回到缩放后尺寸 |

**关键验证**：
- ❌ **旧行为**：需要按 50 次 `Ctrl+Z` 才能回到缩放前
- ✅ **新行为**：只需要按 **1 次** `Ctrl+Z` 就回到缩放前

### 场景 4：缩放过程中 history 长度不变验证

#### 测试步骤

1. **准备测试环境**
   - 启动开发服务器 `npm run dev`
   - 访问编辑器页面 `/builder`
   - 添加一个新组件（确保 history 有一个初始状态）

2. **获取初始 history 长度**
   - 打开开发者工具 → Console
   - 执行以下命令查看当前 history 长度：
   ```javascript
   const { useBuilderStore } = window;
   const state = useBuilderStore.getState();
   console.log('初始 history 长度:', state.history.length);
   console.log('初始 currentIndex:', state.currentIndex);
   console.log('初始 canUndo:', state.canUndo);
   ```
   - 记录初始长度值（记为 `initialLength`）

3. **开始缩放但不结束**
   - 选中组件
   - 将鼠标移到右下角缩放手柄
   - **按住鼠标左键但不松开**（保持 mousemove 状态）
   - 缓慢拖动鼠标，观察组件尺寸变化

4. **验证缩放过程中 history 长度不变**
   - 在按住鼠标的同时，在控制台继续执行：
   ```javascript
   const state = useBuilderStore.getState();
   console.log('缩放中 history 长度:', state.history.length);
   console.log('缩放中 isInHistoryBatch:', state.isInHistoryBatch);
   ```
   - 观察：
     - `isInHistoryBatch` 应该为 `true`
     - `history.length` 应该等于 `initialLength`（没有增长）
     - `canUndo` 应该保持不变

5. **结束缩放**
   - 松开鼠标左键
   - 在控制台执行：
   ```javascript
   const state = useBuilderStore.getState();
   console.log('缩放后 history 长度:', state.history.length);
   console.log('缩放后 isInHistoryBatch:', state.isInHistoryBatch);
   console.log('缩放后 canUndo:', state.canUndo);
   ```
   - 观察：
     - `isInHistoryBatch` 应该为 `false`
     - `history.length` 应该等于 `initialLength + 1`（只增加 1 条）
     - `canUndo` 应该为 `true`

|-----|-------|-------|
| 初始 | `history.length` | N |

| 缩放中 | `canUndo` | 不变 |
| 缩放后 | `isInHistoryBatch` | `false` |
| 缩放后 | `history.length` | N + 1（只增加 1） |
| 缩放后 | `canUndo` | `true` |

**关键验证**：
- ❌ **旧行为**：每次 mousemove 都增加 1 条历史，长度变为 N + 50+
- ✅ **新行为**：缩放过程中长度不变，结束时只增加 1 条

## 6. 技术实现要点

### 6.1 修复原则

1. **批量模式标记**：使用 `isInHistoryBatch` 标记是否处于批量操作模式
2. **状态快照保存**：`beginHistoryBatch` 时保存当前组件状态快照
3. **静默更新**：批量模式下 `pushHistory` 直接返回，不记录历史
4. **单次记录**：`endHistoryBatch` 时比较开始和结束状态，只记录 **1 条** 历史
5. **可选取消**：`cancelHistoryBatch` 取消批量模式，不记录历史

### 6.2 关键代码对比

**修复前（问题代码）**：

```typescript
// useResize.ts
const handleResizeStart = (e, handle) => {
  // 设置状态...
  // 没有批量保护
};

const handleResizeEnd = () => {
  // 重置状态...
  // 没有结束批量
};

// 每次 mousemove 都调用 updateComponent
const handleResizeMove = (e) => {
  updateComponent(id, { x: newX, y: newY, width: newWidth, height: newHeight });
  // 每次都 pushHistory ❌
};
```

**修复后（正确代码）**：

```typescript
// useResize.ts
const handleResizeStart = (e, handle) => {
  // 设置状态...
  beginHistoryBatch();  // ✅ 开始批量模式
};

const handleResizeEnd = () => {
  // 重置状态...
  endHistoryBatch();    // ✅ 结束批量模式，只记录 1 条
};

const handleResizeCancel = () => {
  // 重置状态...
  cancelHistoryBatch();  // ✅ 取消，不记录
};

// 批量模式下，updateComponent 不会 pushHistory
const handleResizeMove = (e) => {
  updateComponent(id, { x: newX, y: newY, width: newWidth, height: newHeight });
  // isInHistoryBatch = true，pushHistory 直接返回 ✅
};
```

### 6.3 API 设计说明

| API | 调用时机 | 作用 |
|-----|---------|------|
| `beginHistoryBatch()` | 操作开始时（mousedown） | 保存当前状态快照，设置批量标记 |
| `endHistoryBatch()` | 操作正常结束时（mouseup） | 比较开始和结束状态，有变化则记录 1 条历史 |
| `cancelHistoryBatch()` | 操作取消时（Escape 或拖拽到无效区域） | 清除批量标记，不记录历史 |

**设计考量**：
1. `beginHistoryBatch` 必须在 **每次操作开始** 时调用
2. `endHistoryBatch` 或 `cancelHistoryBatch` 必须在 **每次操作结束** 时调用
3. 两者必须配对使用，否则 `isInHistoryBatch` 会一直为 `true`
