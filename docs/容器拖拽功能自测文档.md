# 容器拖拽功能自测文档

## 一、功能概述

容器拖拽功能是低代码编辑器的核心功能之一，允许用户在容器内和容器间拖拽组件。该功能基于 `@dnd-kit` 库实现，主要包括：

1. **容器内排序**：使用 `@dnd-kit/sortable` 的 `SortableContext` 实现子组件的拖拽排序
2. **跨容器拖拽**：支持将组件从一个容器拖到另一个容器
3. **嵌套容器支持**：支持多层嵌套容器间的拖拽操作
4. **撤销/重做**：所有拖拽操作都会保存历史记录，支持撤销和重做

## 二、测试场景

### 场景 1：将组件拖入空 Container

**测试步骤**：
1. 在画布上添加一个空的 Container 组件
2. 从组件库拖拽一个 Text 组件到 Container 内部
3. 或者从画布根级别拖拽一个已有组件到 Container 内部

**预期效果**：
- ✅ Text 组件应该被添加到 Container 的 children 数组中
- ✅ Container 的 drop zone 区域在拖拽经过时应该高亮显示
- ✅ 组件不再出现在画布根级别

**异常场景**：
- ❌ Container 为空时无法放入组件 → 检查 `ContainerDropZone` 的 `useDroppable` 配置
- ❌ 组件放入后位置不正确 → 检查 `addComponentToParent` 方法

---

### 场景 2：将组件拖入已有子组件的 Container

**测试步骤**：
1. 创建一个 Container，并在其中放入 2 个组件（Text A, Text B）
2. 从组件库拖拽一个新组件（Button）到 Container 内部
3. 观察新组件的插入位置

**预期效果**：
- ✅ Container 的 children 数组应该有 3 个组件
- ✅ 新组件应该被添加到 children 数组的末尾（或根据拖拽位置插入）
- ✅ UI 上应该能看到 3 个组件按顺序排列

**异常场景**：
- ❌ 新组件覆盖了原有组件 → 检查 `addComponentToParent` 是否正确合并数组
- ❌ 组件数量不正确 → 检查 `add` 操作是否正确

---

### 场景 3：Container 内子组件排序交换位置

**测试步骤**：
1. 创建一个 Container，放入 3 个组件（顺序：A, B, C）
2. 拖拽组件 C 到 A 的前面
3. 观察组件顺序变化

**预期效果**：
- ✅ 拖拽过程中其他组件应该自动让位，显示插入位置
- ✅ 拖拽结束后顺序应该变为：C, A, B
- ✅ Container 的 children 数组顺序应该与 UI 一致
- ✅ 应该可以撤销这个排序操作

**异常场景**：
- ❌ 拖拽时组件不移动 → 检查 `SortableContext` 和 `useSortable` 的配置
- ❌ 顺序变更后数据不同步 → 检查 store 中的数据更新逻辑

---

### 场景 4：将组件从 Container 拖出到画布根区域

**测试步骤**：
1. 创建一个 Container，放入一个 Text 组件
2. 选中 Text 组件，拖拽到 Container 外面的画布空白区域
3. 释放鼠标

**预期效果**：
- ✅ Text 组件应该从 Container 的 children 中移除
- ✅ Text 组件应该出现在画布根级别（components 数组中）
- ✅ Container 现在应该显示为空状态
- ✅ 应该可以撤销这个操作

**异常场景**：
- ❌ 组件同时存在于 Container 和根级别 → 检查 `extractComponentFromTree` 逻辑
- ❌ 组件从 Container 移除但没有添加到根级别 → 检查 `moveComponentToParent` 的 null 处理

---

### 场景 5：将组件从一个 Container 拖入另一个 Container

**测试步骤**：
1. 创建两个 Container：Container A 和 Container B
2. 在 Container A 中放入一个 Text 组件
3. 将 Text 组件从 Container A 拖拽到 Container B 内部
4. 释放鼠标

**预期效果**：
- ✅ Text 组件应该从 Container A 的 children 中移除
- ✅ Text 组件应该被添加到 Container B 的 children 中
- ✅ Container A 现在应该为空（或显示空状态）
- ✅ Container B 现在应该有 1 个子组件
- ✅ 应该可以撤销这个操作

**异常场景**：
- ❌ 组件在两个 Container 中都存在 → 检查 `extractComponentFromTree` 是否正确移除
- ❌ 组件没有被添加到目标 Container → 检查目标 Container 的 drop zone 是否正确识别

---

### 场景 6：2层嵌套容器拖拽（Container A ⊃ Container B ⊃ 子组件）

**测试步骤 6.1：从内层拖到外层**：
1. 创建嵌套结构：Container A 包含 Container B，Container B 包含 Text 组件
2. 将 Text 组件从 Container B 拖拽到 Container A 中（但不进入 Container B）
3. 释放鼠标

**预期效果 6.1**：
- ✅ Text 组件应该从 Container B 的 children 中移除
- ✅ Text 组件应该被添加到 Container A 的 children 中
- ✅ Container A 现在应该有 2 个子组件：Container B 和 Text
- ✅ Container B 现在应该为空

**测试步骤 6.2：从外层拖到内层**：
1. 确保 Container A 包含 Container B 和 Text 组件
2. 将 Text 组件从 Container A 拖拽到 Container B 内部
3. 释放鼠标

**预期效果 6.2**：
- ✅ Text 组件应该从 Container A 的 children 中移除
- ✅ Text 组件应该被添加到 Container B 的 children 中
- ✅ Container A 现在应该只有 1 个子组件：Container B
- ✅ Container B 现在应该有 1 个子组件：Text

**测试步骤 6.3：从内层拖到根级别**：
1. 确保 Container A 包含 Container B，Container B 包含 Text 组件
2. 将 Text 组件从 Container B 拖拽到画布根级别（Container A 外面）
3. 释放鼠标

**预期效果 6.3**：
- ✅ Text 组件应该从 Container B 的 children 中移除
- ✅ Text 组件应该出现在画布根级别
- ✅ Container B 现在应该为空

**异常场景**：
- ❌ 嵌套层级识别错误 → 检查 `findComponentById` 和 `extractComponentFromTree` 的递归逻辑
- ❌ 父容器更新错误 → 检查 `addComponentToParentInTree` 的递归逻辑

### 场景 7：拖拽操作后的撤销/重做

**测试步骤**：
1. 创建初始状态：空画布
2. 操作 1：添加一个 Container（可撤销）
3. 操作 2：将一个 Text 组件拖入 Container（可撤销）
4. 操作 3：将 Text 组件拖出到根级别（可撤销）
5. 连续撤销 3 次
6. 连续重做 3 次

**预期效果**：
- ✅ 每次操作后，`canUndo` 应该为 true
- ✅ 撤销 1 次：Text 组件应该回到 Container 内
- ✅ 撤销 2 次：Text 组件应该从 Container 中移除（Container 为空）
- ✅ 撤销 3 次：Container 应该被移除，回到空画布
- ✅ 重做 3 次后，应该回到操作 3 完成后的状态
- ✅ 每次撤销/重做后，UI 应该与数据同步

**异常场景**：
- ❌ 撤销后数据不一致 → 检查 `pushHistory` 是否在正确的时机被调用
- ❌ 重做不起作用 → 检查 history 数组和 currentIndex 的管理
- ❌ 嵌套组件的历史记录不正确 → 检查 `structuredClone` 是否正确处理嵌套结构

### 场景 8：边界情况测试

**测试步骤 8.1：将 Container 拖入另一个 Container**：
1. 创建两个 Container：A 和 B
2. 将 Container B 拖入 Container A 内部

**预期效果 8.1**：
- ✅ Container B 应该成为 Container A 的子组件
- ✅ Container B 的原有子组件应该保持不变
- ✅ 不应该产生异常

**测试步骤 8.2：拖拽到非容器区域**：
1. 创建一个 Container，放入一个 Text 组件
2. 将 Text 组件拖拽到画布外（浏览器窗口边缘）
3. 释放鼠标

**预期效果 8.2**：
- ✅ 不应该产生异常
- ✅ Text 组件应该保持在原来的位置（取消拖拽）
- ✅ 数据状态不应该改变

**测试步骤 8.3：将 Container 拖入自身**：
1. 创建一个 Container
2. 尝试将这个 Container 拖拽到它自己的 drop zone 中

**预期效果 8.3**：
- ✅ 应该被检测并阻止
- ✅ 不应该产生异常
- ✅ Container 状态不应该改变

**异常场景**：
- ❌ 拖拽到非容器区域导致数据丢失 → 检查 `handleDragEnd` 中的边界条件

## 五、常见问题排查

### 问题 1：拖拽时组件不跟随鼠标移动
**可能原因**：
- `useSortable` 的 `id` 与 `SortableContext` 的 `items` 不匹配
- `transform` 没有正确应用到样式

**排查步骤**：
1. 检查 `createSortableItemId` 生成的 ID 是否一致
2. 检查 `CSS.Transform.toString(transform)` 是否正确应用
3. 检查 `SortableContext` 的 `items` 数组是否包含当前拖拽项的 ID

### 问题 2：跨容器拖拽时数据不同步
**可能原因**：
- `extractComponentFromTree` 没有正确从嵌套结构中移除组件
- `addComponentToParentInTree` 没有正确添加到目标位置

**排查步骤**：
1. 打印拖拽前后的 `components` 数组
2. 检查 `findComponentLocation` 返回的位置是否正确
3. 检查 `moveComponentToParent` 中的 `extract` 和 `add` 操作顺序

### 问题 3：撤销/重做后 UI 不更新
**可能原因**：
- `pushHistory` 没有被调用
- Zustand 的状态更新没有触发重渲染

**排查步骤**：
1. 检查每次数据变更操作后是否调用了 `pushHistory`
2. 检查 `undo` 和 `redo` 方法中是否正确设置了 `components` 状态
3. 确保使用 `structuredClone` 进行深拷贝

### 问题 4：嵌套容器拖拽层级错误
**可能原因**：
- 递归函数没有正确处理所有层级
- 父容器 ID 识别错误

**排查步骤**：
1. 检查 `findComponentById` 的递归逻辑
2. 检查 `extractComponentFromTree` 是否在每一层都正确查找
3. 打印嵌套结构的完整路径

## 六、性能优化建议

1. **避免不必要的重渲染**：
   - 使用 `React.memo` 包裹 `SortableContainerChild`
   - 使用 `useMemo` 缓存 sortableIds 数组

2. **历史记录管理**：
   - `MAX_HISTORY_LENGTH` 设置为合理值（当前 50）
   - 避免在短时间内大量操作导致内存占用过高

3. **深拷贝优化**：
   - 当前使用 `structuredClone`，对于大型组件树可能有性能影响
   - 可以考虑使用 immer 或其他 immutable 库
